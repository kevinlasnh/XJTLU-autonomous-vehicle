## 2026.05.08

### [GPS Corridor Deployment Fixes and v3 On-Vehicle Closeout]

#### Context

This round verified the deployability of the GPS Corridor stack on the physical vehicle and fixed two engineering issues found in previous tests:

- The route runner explicitly cancelled the current Nav2 goal during live alignment updates, creating a short stop before the replacement goal was sent.
- The first 30m subgoal could fall outside the old 50m rolling global costmap, triggering Nav2 recovery spin and backup.
- Corridor/explore launches could fall back to the simpler `pgo` package RViz layout instead of the full bringup layout.

#### Deployment Commits

- `4e04e88` `Prevent corridor alignment preempt stalls`
- `83a0bbe` `Expand corridor costmap and restore RViz layout`

#### Key Changes

1. `gps_route_runner_node.py`
   - Removed explicit `cancel_goal_async()` + `spin_until_future_complete()` from the normal alignment-shift correction path.
   - When alignment shift exceeds the threshold, the runner now returns a sentinel, recomputes the subgoal, and immediately sends a new `NavigateToPose` goal.
   - Nav2 native goal preemption now owns the old-goal transition, avoiding a no-goal gap between cancel and re-send.
   - Added `alignment_shift_cooldown_s: 3.0` to prevent high-frequency preemption caused by ordinary GPS jitter.

2. `master_params.yaml`
   - Added `alignment_shift_cooldown_s: 3.0`.
   - Updated comments from cancel/recompute wording to preempt/recompute wording.

3. `nav2_explore.yaml`
   - `global_costmap.width: 50 -> 100`
   - `global_costmap.height: 50 -> 100`
   - Purpose: increase rolling global costmap radius from about 25m to about 50m, covering the current 30m corridor subgoal.

4. `system_explore.launch.py`
   - Added an `rviz_config` launch argument.
   - Defaulted it to `bringup/rviz/pgo.rviz` to restore the full visualization layout.

#### Final Assessment

This deployment round can be closed:

- The alignment preemption mechanism triggered on the physical vehicle.
- The 100x100 global costmap removed the direct 30m subgoal off-map failure.
- RViz now points back to the full bringup layout.
- PGO/FAST-LIO2 continuously received cloud + odom; logs do not indicate odometry, point cloud map, or `map -> odom` divergence.

The remaining issue is not Nav2, costmap, PGO, or FAST-LIO2 divergence. It is the absolute positioning and ENU -> map anchoring/conversion accuracy of ordinary GNSS `/fix`. The user's observation that the converted GPS coordinates do not correspond to the correct physical points in Nav2 is consistent with the repeated alignment corrections in the logs.

#### Next Direction

No further runner or YAML parameter changes were made in this closeout. The next phase should treat this as a GNSS/RTK localization accuracy task and evaluate:

- RTK GNSS receiver and correction-source integration.
- `robot_localization` or an equivalent fusion chain for GNSS/IMU/FAST-LIO2 pose quality management.
- Stricter GPS quality gates, so ordinary fix data is not treated as physically precise enough for decimeter-level routing.
- Map-anchored physical waypoint routes, multi-point rigid registration, or continuous trajectory collection instead of single-point `start_ref` anchoring.
