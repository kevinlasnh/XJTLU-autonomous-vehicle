## 2026.05.08

### [GPS Corridor 部署修复与 v3 实车测试收口]

#### 背景

本轮目标是验证 GPS Corridor 在实车中的部署链路，并修复前序测试暴露的两个工程问题：

- live alignment 更新时，route runner 显式取消旧 Nav2 goal，造成短暂停车。
- 第一个 30m subgoal 超出 50m rolling global costmap 的 25m 半径，引发 Nav2 recovery 中的旋转与后退。
- corridor/explore 启动时 RViz 落回 `pgo` 包内较简单的默认面板，未使用 bringup 中的完整布局。

#### 部署提交

- `4e04e88` `Prevent corridor alignment preempt stalls`
- `83a0bbe` `Expand corridor costmap and restore RViz layout`

#### 关键改动

1. `gps_route_runner_node.py`
   - 删除 alignment shift 正常修正路径里的显式 `cancel_goal_async()` + `spin_until_future_complete()`。
   - alignment shift 超阈值时直接返回 sentinel，让外层立即重算 subgoal 并发送新的 `NavigateToPose` goal。
   - 由 Nav2 原生 goal preemption 接管旧 goal，避免 cancel 与 re-send 之间的零目标间隙。
   - 新增 `alignment_shift_cooldown_s: 3.0`，防止普通 GPS 抖动导致高频 preempt。

2. `master_params.yaml`
   - 新增 `alignment_shift_cooldown_s: 3.0`，并把注释从 cancel/recompute 调整为 preempt/recompute。

3. `nav2_explore.yaml`
   - `global_costmap.width: 50 -> 100`
   - `global_costmap.height: 50 -> 100`
   - 目的：rolling global costmap 半径从约 25m 扩到约 50m，覆盖当前 30m corridor subgoal。

4. `system_explore.launch.py`
   - 新增 `rviz_config` launch argument。
   - 默认传入 `bringup/rviz/pgo.rviz`，恢复完整可视化面板。

#### 最终判断

本轮部署修复可以收口：

- alignment preempt 机制已在实车中触发。
- 100x100 global costmap 消除了 30m subgoal 直接越界的问题。
- RViz 配置路径已恢复到 bringup 完整面板。
- PGO/FAST-LIO2 持续收到 cloud + odom，日志未显示里程计、点云地图或 `map -> odom` 发散。

剩余问题不是 Nav2、costmap、PGO 或 FAST-LIO2 发散，而是普通 GNSS `/fix` 的绝对坐标及 ENU -> map 锚定/换算精度不足。用户实车观察到的“GPS 转换后的坐标在 Nav2 中不是正确物理点坐标”与日志中的持续 alignment 修正一致。

#### 后续方向

本轮不再继续修改当前 runner 或 YAML 参数。后续应作为新的 GNSS/RTK 定位精度任务进入方案阶段，重点评估：

- RTK GNSS 接收机与改正源接入。
- `robot_localization` 或等价融合链路，把 GNSS/IMU/FAST-LIO2 位姿质量统一管理。
- 更严格的 GPS quality gate，避免普通 fix 被误认为可用于厘米/分米级物理路径。
- map 物理点路线、多点刚体配准或连续轨迹采集，替代单点 `start_ref` 锚定。
