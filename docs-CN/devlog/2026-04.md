### [GPS Live Alignment + 提速到 1.5 m/s]

针对上午 GPS corridor 测试暴露的对齐问题，晚间完成四项核心改动并通过室内高速验证。

#### Change 1: Calibration Translation-Only (`ebc26e2`)

- **文件**: `gps_global_aligner_node.py`
- **改动**: `_solve_calibration_alignment()` 移除完整 2D 旋转求解（centroid + H 矩阵 + atan2），改为固定 bootstrap theta、仅估计平移
- **原因**: 点对少且空间分布差时，旋转求解会出现 ~180° 翻转（如上午测试中的 176.95deg delta）
- **效果**: 避免 waypoint calibration 旋转翻转，保持旋转稳定性

#### Change 2: 启动 GPS Offset 吸收 (`fe3933e`)

- **文件**: `gps_global_aligner_node.py`
- **改动**: `_build_bootstrap_alignment()` 新增 `startup_fix` 参数，优先使用上电首个 stable GPS 作为 ENU 锚点，而非路线文件中的 `start_ref`
- **原因**: 上午测试中 startup GPS offset ~4.11m 未被吸收，导致第一段目标系统性偏移
- **效果**: 启动时直接对齐到当天实际 GPS 位置，消除启动偏移

#### Change 3: Runner Live Alignment (`e73c2bf`)

- **文件**: `gps_route_runner_node.py`（核心改动）
- **改动**:
  - 移除 per-waypoint frozen alignment 机制

  - 删除 waypoint calibration 请求/等待/守卫相关代码（~200 行）
- **原因**: 上午测试中 per-waypoint frozen alignment 导致第二段 live alignment 被 guard 拒绝（4.96m > 3.0m），runner 回退到旧 alignment，目标持续偏左

## 2026.04.02

### [MPPI 直线稳定性与旋转速度联调]

针对 2026-04-01 晚间测试发现的"无障碍直线路径蛇形修正"问题，完成多轮 MPPI 参数调优，最终通过用户室内实测。

#### 第一轮：收紧采样参数 (`e06ec03`)

- **Commit**: `e06ec03` `Dampen MPPI straight-line oscillation at high speed`
- **改动**: 收紧 `vx_std`, `PathAlign`, `PathFollow`, `PathAngle` 参数
- **用户反馈**: 仍有残余蛇形
- **结论**: 单纯收紧采样不足以解决问题，需要从路径源头处理

#### 第二轮：引入路径平滑链 (`86c3cb8`)

- **Commit**: `86c3cb8` `Smooth NavFn paths and stabilize MPPI straight tracking`
- **调研发现**:
  - Nav2 官方文档明确指出 `Savitzky-Golay smoother` 特别适合去掉 `NavFn` 路径中的细小折线和锯齿
  - 仅配置 `smoother_server` 不会自动生效，必须在 BT 中显式插入 `SmoothPath`
- **改动**:
  - BT XML: 在 `ComputePathToPose` 后插入 `SmoothPath`，使用 `savitzky_golay_smoother`
  - `nav2_explore.yaml`: 将 `simple_smoother` 替换为 `savitzky_golay_smoother`
- **效果**: 消除 NavFn 原始路径的细碎毛刺，为 MPPI 提供更平滑的输入

#### 第三轮：提高角速度上限 (`ed2694e`)

- **Commit**: `ed2694e` `Raise angular speed limits to match 1.5 m/s tracking`
- **改动**:
  - `wz_max`: 1.0 → 1.75 rad/s
  - `az_max`: 3.5 → 6.0 rad/s²
- **目标**: 提高旋转灵活性，匹配 1.5 m/s 直线速度的转弯需求

#### 第四轮：增加角度灵活性 (`4c2466d`)

- **Commit**: `4c2466d` `Increase angular agility and tighten post-avoidance path return`
- **改动**:
  - `wz_std`: 0.3 → 0.35
  - `PathAlignCritic.cost_weight`: 8.0 → 11.0
  - `PathFollowCritic.cost_weight`: 8.0 → 6.0
  - `PathAngleCritic.cost_weight`: 2.0 → 4.0
- **副作用**: 提高了旋转灵活性，但把直线稳定性重新拉坏
- **结论**: 需要回滚直线相关参数，仅保留角速度上限增强

#### 第五轮：回滚不稳定参数 (`e9b23df`)

- **Commit**: `e9b23df` `Revert unstable straight-tracking tweaks while keeping angular gains`
- **改动**: 回滚 `4c2466d` 中的 `wz_std` 和 critic 权重调整，保留 `ed2694e` 的角速度上限增强
- **效果**: 恢复直线稳定性，同时保持高角速度能力

#### 第六轮：最终收口 (`15d4cac`)

- **Commit**: `15d4cac` `Reduce residual MPPI straight-line oscillation`
- **调研依据**: Nav2 官方文档推荐 `PathAlignCritic.offset_from_furthest` 经验公式为 `prediction_horizon_s * max_speed / path_resolution / 3`
- **改动**:
  - `wz_std`: 0.25 → 0.22（略收角速度采样）
  - `PathAlignCritic.cost_weight`: 11.0 → 12.0（更早介入路径对齐）
  - `PathAlignCritic.offset_from_furthest`: 24 → 14（更早启用对齐 critic）
  - `PathFollowCritic.threshold_to_consider`: 0.6 → 1.4（延后退出跟随 critic）
- **原理**: 原先 `offset_from_furthest=24` 偏大，属于"晚介入"，容易在纯直线高速下出现轻微左右修正；降低到 14 后更早启用对齐，配合延后退出跟随，形成更稳定的直线跟踪

#### 用户实测结论

- **测试模式**: 室内导航（`indoor-nav`）
- **最终评价**: "相当不错，已经可以完美成功"
- **通过标准**: 按"通过并收口"处理
- **当前状态**: 直线稳定性问题已解决，交回 CC 文档阶段

#### 技术总结

本轮调优的核心发现：
1. **路径源头处理优先**: NavFn 原始路径的细碎折线会直接影响 MPPI 跟踪稳定性，必须在 BT 中显式插入 `SmoothPath`
2. **MPPI 能力边界**: Nav2 官方明确把"exact path following"归给 RPP，MPPI 的目标是在不牺牲动态避障的前提下尽量减少蛇形
3. **Critic 介入时机**: `offset_from_furthest` 是关键参数，应根据预测时域和速度计算，而非盲目调大
4. **参数组合验证**: 单独提高角速度上限是安全的，但同时调整 critic 权重可能破坏直线稳定性，需要分步验证

### [动态避障：恢复行为与全局重规划优化]

- **日期**: 2026-04-05
- **背景**: 动态障碍物响应速度慢，现有 3 级恢复机制在遇到活动障碍物时过于激进（容易直接全清代价地图或后退）。
- **重规划频率提升**:
  - 行为树 (`navigate_to_pose_w_replanning_3hz_and_recovery.xml`) 中的全局重规划频率从 3.0Hz 提升至 5.0Hz。
  - `nav2_explore.yaml` 中 `expected_planner_frequency` 同步提至 5.0。
  - 减小全局代价地图尺寸以降低计算开销：`width`/`height` 从 80x80 缩小到 50x50。
  - 全局代价地图 `update_frequency` 从 3.0 提至 5.0，`publish_frequency` 从 1.5 提至 2.0。
- **NavfnPlanner 优化**: 启用 A* 搜索 (`use_astar: true`) 以替代默认的 Dijkstra，提高 5Hz 规划下的搜索效率。
- **进度检查器收紧**: `movement_time_allowance` 从 15.0s 缩短至 10.0s，更快判定卡死。
- **5 级渐进式恢复机制**:
  1. **局部保留清理** (`ClearCostmapExceptRegion`, reset_distance=2.0)：保留近处真实障碍物，仅清除远处的陈旧数据。
  2. **等待** (`Wait`, 2s)：等待 STVL 体素自然衰减（voxel_decay 为 0.8s）。
  3. **原地旋转** (`Spin`, 1.57 rad)：旋转 90 度以使用 LiDAR 重新扫描周围环境。
  4. **全清代价地图** (`ClearEntireCostmap`)：核弹级重置，清除全部局部和全局代价地图。
  5. **物理后退** (`BackUp`, 1.0m, 0.15m/s)：最终手段，物理后退 1 米（`fec59f3` 调整）。
