## Difference Between Trajectory and Path
1. Trajectory and path are two completely different concepts in robot navigation, although they may appear similar.
2. Path
   1. A collection of geometric points in space
   2. Contains position [x,y] information, may also include orientation
   3. Physical meaning: **this is the route I want to take**
   4. Generated by a planner, a kind of global path planner
   5. Constraints: static obstacles and connectivity
3. Trajectory
   1. A collection of spatial points + time/velocity information
   2. Contains position [x,y,theta] and velocity [v,w] information
   3. Physical meaning: at what time and at what speed should I pass through here
   4. Generated by a local controller, such as the DWB controller
   5. Constraints: robot dynamics constraints (max velocity, acceleration), dynamic obstacles
4. A path does not care how fast you are driving or whether you are drifting, but a trajectory is a specific driving plan that includes not only the route but all detailed velocity information, explicitly specifying the robot's state at every point in time.

5. Use `# >>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>>> ` inline comments to highlight adjusted parameters in YAML files; comments above parameters are explanatory.
6. Separate large parameter blocks in Nav2 YAML files with three blank lines.
7. After adjusting parameters, always run the system to verify vehicle functionality is intact. Do not adjust parameters and wait days before testing!
8. `map_server` loads and publishes pre-created static map data, providing the static obstacle layer for costmaps.
9. AMCL (Adaptive Monte Carlo Localization) uses particle filter algorithm based on LiDAR data and maps for real-time robot position estimation. Input: `/scan`, `/map`, and frontend odometry TF chain (`odom -> base_link`). Output: backend odometry TF chain (`map -> odom`) and robot position `/amcl_pose`. It can only be used for localization, not mapping, and is suitable for navigation in known maps.

## Nav2 Target Pose Format
> ---
> header:
>   stamp: <timestamp>
>   frame_id: "map"   # Usually the map coordinate frame (fixed map frame)
> pose:
>   position:
>     x: 2.5          # Target x coordinate (unit: meters)
>     y: -1.0         # Target y coordinate (unit: meters)
>     z: 0.0          # Usually 0 (2D navigation)
>   orientation:
>     x: 0.0
>     y: 0.0
>     z: 0.7071       # Quaternion-represented yaw angle
>     w: 0.7071       # e.g., facing east (90 degrees), equivalent to yaw = 1.57 rad
> ---

4. Studied the difference between poses and paths.
5. Current costmap development overview:
   1. Current goal is to navigate using a single costmap -- the local costmap, which is a rolling window map surrounding the vehicle, i.e., the local costmap.
   2. For global path planning, we do not need a global costmap. We only need the global coordinate frame, our start pose, goal pose, and all waypoints. For global path analysis, we should view the path and point-cloud-based obstacles as two separate layers. First, the map has no obstacles; we only know our start pose, goal pose, and all waypoints. The global path is then a straight line from start to the first waypoint, then from the first to the second waypoint, and so on to the goal. Now adding obstacles: these obstacles are only detected within the vehicle's rolling window. The path between the start and first waypoint changes only when obstacles are detected within the rolling window; otherwise, the default path between any two points is a straight line.
   3. The DWB controller ultimately exists to follow the path.
   4. When encountering complex obstacles, without global path updates the DWB controller may not know which way to go. The classic example is a U-shaped obstacle.
   5. Two approaches: first, create a super-map combining all local and global costmap capabilities; second, have local and global costmaps share the same data source -- this is worth trying.
   6. Dual-map navigation aligns with human brain thinking patterns.
   7. However, there is a critical issue: **in local obstacle avoidance, the local and global costmaps may conflict. With dual maps, within a certain range the global and local costmap obstacle layers overlap. If the local costmap detects an obstacle's disappearance but the global costmap has not refreshed due to lower update rate, the system would try to avoid a non-existent obstacle.** Nav2's system architecture design has corresponding solutions for this, primarily through controller parameter configuration.
