### [GPS Live Alignment + Speed Increase to 1.5 m/s]

In response to alignment issues exposed in morning GPS corridor tests, completed four core changes in the evening and passed indoor high-speed verification.

#### Change 1: Calibration Translation-Only (`ebc26e2`)

- **File**: `gps_global_aligner_node.py`
- **Change**: `_solve_calibration_alignment()` removed full 2D rotation solving (centroid + H matrix + atan2), switched to fixed bootstrap theta with translation-only estimation
- **Reason**: With few point pairs and poor spatial distribution, rotation solving produces ~180° flips (e.g., 176.95deg delta in morning test)
- **Effect**: Avoids waypoint calibration rotation flips, maintains rotation stability

#### Change 2: Startup GPS Offset Absorption (`fe3933e`)

- **File**: `gps_global_aligner_node.py`
- **Change**: `_build_bootstrap_alignment()` added `startup_fix` parameter, prioritizes using first stable GPS at power-on as ENU anchor instead of `start_ref` from route file
- **Reason**: Morning test showed startup GPS offset ~4.11m not absorbed, causing systematic offset in first segment targets
- **Effect**: Aligns to actual GPS position on startup day, eliminates startup offset

#### Change 3: Runner Live Alignment (`e73c2bf`)

- **File**: `gps_route_runner_node.py` (core change)

  - Removed per-waypoint frozen alignment mechanism

- **Reason**: Morning test showed per-waypoint frozen alignment caused second segment live alignment rejection (4.96m > 3.0m guard), runner fell back to old alignment, targets continued left offset
- **Effect**: Navigation targets automatically update with GPS real-time refresh, no need to wait for waypoint boundaries

## 2026.04.02

### [MPPI Straight-Line Stability and Rotation Speed Joint Tuning]

Addressing the "clear straight path serpentine correction" issue discovered in 2026-04-01 evening tests, completed multiple rounds of MPPI parameter tuning, ultimately passing user indoor on-vehicle testing.

#### Round 1: Tighten Sampling Parameters (`e06ec03`)

- **Commit**: `e06ec03` `Dampen MPPI straight-line oscillation at high speed`
- **Changes**: Tightened `vx_std`, `PathAlign`, `PathFollow`, `PathAngle` parameters
- **User Feedback**: Residual serpentine still present
- **Conclusion**: Simply tightening sampling insufficient to solve the problem, need to address path source

#### Round 2: Introduce Path Smoothing Chain (`86c3cb8`)

- **Commit**: `86c3cb8` `Smooth NavFn paths and stabilize MPPI straight tracking`
- **Research Findings**:
  - Nav2 official documentation explicitly states `Savitzky-Golay smoother` is particularly suitable for removing small zigzags and jaggedness in `NavFn` paths
  - Simply configuring `smoother_server` does not automatically take effect, must explicitly insert `SmoothPath` in BT
- **Changes**:
  - BT XML: Insert `SmoothPath` after `ComputePathToPose`, using `savitzky_golay_smoother`
  - `nav2_explore.yaml`: Replace `simple_smoother` with `savitzky_golay_smoother`
- **Effect**: Eliminate fine burrs in NavFn raw paths, provide smoother input for MPPI

#### Round 3: Raise Angular Speed Limits (`ed2694e`)

- **Commit**: `ed2694e` `Raise angular speed limits to match 1.5 m/s tracking`
- **Changes**:
  - `wz_max`: 1.0 → 1.75 rad/s
  - `az_max`: 3.5 → 6.0 rad/s²
- **Goal**: Increase rotational agility to match turning requirements at 1.5 m/s linear speed

#### Round 4: Increase Angular Agility (`4c2466d`)

- **Commit**: `4c2466d` `Increase angular agility and tighten post-avoidance path return`
- **Changes**:
  - `wz_std`: 0.3 → 0.35
  - `PathAlignCritic.cost_weight`: 8.0 → 11.0
  - `PathFollowCritic.cost_weight`: 8.0 → 6.0
  - `PathAngleCritic.cost_weight`: 2.0 → 4.0
- **Side Effect**: Improved rotational agility but degraded straight-line stability again
- **Conclusion**: Need to revert straight-line related parameters, keep only angular speed limit enhancements

#### Round 5: Revert Unstable Parameters (`e9b23df`)

- **Commit**: `e9b23df` `Revert unstable straight-tracking tweaks while keeping angular gains`
- **Changes**: Reverted `wz_std` and critic weight adjustments from `4c2466d`, kept angular speed limit enhancements from `ed2694e`
- **Effect**: Restored straight-line stability while maintaining high angular speed capability

#### Round 6: Final Closeout (`15d4cac`)

- **Commit**: `15d4cac` `Reduce residual MPPI straight-line oscillation`
- **Research Basis**: Nav2 official documentation recommends `PathAlignCritic.offset_from_furthest` empirical formula as `prediction_horizon_s * max_speed / path_resolution / 3`
- **Changes**:
  - `wz_std`: 0.25 → 0.22 (slightly reduce angular velocity sampling)
  - `PathAlignCritic.cost_weight`: 11.0 → 12.0 (earlier path alignment intervention)
  - `PathAlignCritic.offset_from_furthest`: 24 → 14 (earlier activation of alignment critic)
  - `PathFollowCritic.threshold_to_consider`: 0.6 → 1.4 (delayed exit from follow critic)
- **Rationale**: Original `offset_from_furthest=24` was too large, representing "late intervention", prone to slight left-right corrections on pure straight lines at high speed; reducing to 14 enables earlier alignment activation, combined with delayed follow exit, forming more stable straight-line tracking

#### User On-Vehicle Test Conclusion

- **Test Mode**: Indoor navigation (`indoor-nav`)
- **Final Evaluation**: "Quite good, can already succeed perfectly"
- **Pass Criteria**: Closed as "passed"
- **Current Status**: Straight-line stability issue resolved, handed back to CC for documentation phase

#### Technical Summary

Core findings from this tuning round:
1. **Path Source Processing Priority**: Fine zigzags in NavFn raw paths directly affect MPPI tracking stability, must explicitly insert `SmoothPath` in BT
2. **MPPI Capability Boundary**: Nav2 official documentation explicitly assigns "exact path following" to RPP, MPPI's goal is to minimize serpentine while not sacrificing dynamic obstacle avoidance
3. **Critic Intervention Timing**: `offset_from_furthest` is a key parameter, should be calculated based on prediction horizon and speed, not blindly increased
4. **Parameter Combination Validation**: Raising angular speed limits alone is safe, but simultaneously adjusting critic weights may break straight-line stability, requires step-by-step verification
