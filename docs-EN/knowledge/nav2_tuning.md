### Known Improvements Needed

- The costmap canvas itself is no longer the main bottleneck; the real limits come from rolling-window semantics, local visibility, and scene geometry
- `5 Hz` replanning improves dynamic-obstacle response, but large-scale outdoor rerouting capability still needs more field validation
- The current MPPI baseline prioritizes stable path tracking while retaining dynamic obstacle avoidance, rather than pure high-speed corridor tuning

## 6. GPS-Specific Configuration (`nav2_gps.yaml`)

GPS goal navigation mode does not modify `nav2_explore.yaml` directly; instead, a separate `nav2_gps.yaml` is created.

Minimum necessary differences from the Explore configuration:

- `general_goal_checker.xy_goal_tolerance = 3.0`
- `general_goal_checker.yaw_goal_tolerance = 0.5`
- `GridBased.tolerance = 2.5`
- `BaseObstacle.scale = 0.02`
- `GoalAlign.scale = 24.0`
- `RotateToGoal.scale = 32.0`

Tuning principles:
- Relax goal tolerance in low-accuracy GNSS environments
- Keep the existing DWB / costmap main structure unchanged
- Do not introduce larger changes like MPPI or VoxelLayer in the GPS MVP branch

## 7. Current Operational Notes (2026-04)

1. RViz fixed frame must be set to `map`.
2. If `map -> odom` is not established, even with Livox and FAST-LIO2 running, RViz may appear blank or the costmap may not display.
3. Explore and Corridor modes are now unified under the MPPI controller (commit `9d71823`).
4. `velocity_smoother.max_velocity[0]` is now `1.0`, matching the anti-understeering baseline absorbed on 2026-04-15.
5. `nav2_gps.yaml` still uses the DWB controller, independent of Explore/Corridor.
6. FAST-LIO2 published point cloud is now height-filtered at the C++ level with window `[-0.33, 0.30]` (commit `f619fa6`); downstream STVL receives clean data.
